# Project Structure

仅当用户明确要求创建或维护分析项目、现有项目已经采用落盘协作，或当前任务明确要求持久化且可追溯的交付物时，才在仓库内创建项目目录。问题复杂、包含资料或可能跨多轮，本身都不足以授权写文件；Quick 和 Standard 模式默认留在对话中。

如果落盘会明显改善后续协作但尚未包含在用户要求中，先说明拟创建的最小范围并请求确认。进入 Project 模式后，默认只创建 `brief.md`，再按实际产物扩展；不要为了“完整”预创建目录、入口文件或状态文件。

## 文件职责

| 文件 | 职责 | 创建条件 |
|---|---|---|
| `brief.md` | 项目总控摘要，记录真实问题、资料状态、当前阶段、下一步 | 每个 Project 项目必需 |
| `decision-map.md` | 复杂项目的路线唯一来源：已决事项、开放问题、迷雾 | 出现多个未决分支或迷雾时 |
| `README.md` | 给人看的项目入口和关键链接 | 需要给团队或读者提供入口时 |
| `AGENT.md` | 给 agent 的稳定规则、上下文入口和工作边界 | 当前运行环境或协作方式需要时 |
| `CLAUDE.md` | 指向 `AGENT.md`，避免两套规则 | 需要相应平台入口时 |

动态状态只维护在 `brief.md` 和按需创建的 `decision-map.md`。`AGENT.md` 不记录当前阶段、资料状态或路线状态。

## 项目根目录

### 最小框架

```text
projects/<YYYYMMDD>-<project-name>/
└── brief.md                 # 项目总控摘要：真实问题、资料状态、当前阶段、下一步
```

### 按需扩展

```text
projects/<YYYYMMDD>-<project-name>/
├── README.md                # 需要面向人提供项目入口时
├── AGENT.md                 # 当前运行环境需要稳定 agent 入口时
├── CLAUDE.md                # 需要相应平台入口时
├── 0-inputs/                # 确有需要持续引用的原始资料时
├── 0-materials/             # 需要整理证据、来源或术语时
├── decision-map.md          # 复杂项目：路线图、已决事项、开放问题和迷雾
├── 1-problem-definition/    # 第 1 步：明确问题、范围、相关方、紧急程度
├── 2-problem-structuring/   # 第 2 步：MECE 逻辑树、鱼骨图、拆解备选方案
├── 3-prioritization/        # 第 3 步：80/20、重要性-紧急性、影响-难度排序
├── 4-work-planning/         # 第 4 步：工作计划、任务分工、时间表、风险
├── 5-analysis/              # 第 5 步：数据分析、根因分析、假设验证
├── 6-synthesis/             # 第 6 步：综合建议、方案评估、方案选择
├── 7-communication/         # 第 7 步：方案呈现、故事线、汇报稿、可视化
├── 8-prompts/               # 最终执行者是 AI/agent 时
└── 9-archive/               # 确需保留废弃版本、旧假设或历史材料时
```

## 命名规则

- 所有项目都直接放在 `projects/` 下，不再按技能类型增加二级分类目录。
- 目录名前缀使用创建日期 `YYYYMMDD`，例如 `20260630-新品类进入策略` 或 `20260630-market-entry-strategy`。
- `<project-name>` 使用简短主题名；中文项目可直接用中文目录名，英文项目用小写连字符。
- 阶段目录按需创建。只有当该阶段产生实质产物时，才创建对应目录。
- 最小项目只包含 `brief.md`；其他文件和目录都必须有当前任务所需的实际内容。
- 阶段文件使用编号前缀保持顺序，例如 `1-problem-definition/v1.md`、`2-problem-structuring/logic-tree-v1.md`。
- 重要版本保留 `v1`、`v2`；不要覆盖已经用于决策的旧版本。
- 原始资料只放入 `0-inputs/`；处理后资料放入 `0-materials/`；分析产物不要和原始资料混放。
- `brief.md` 是项目唯一的总控文件，每次阶段推进后更新当前状态和下一步。
- 当问题大到需要跨多轮拆解时，新增 `decision-map.md` 管理路线；不要在 `brief.md` 或 `AGENT.md` 中重复维护已决事项、开放问题和迷雾。
- 仅在需要相应的人类入口或平台入口时创建 `README.md`、`AGENT.md`、`CLAUDE.md`；不要为了模板完整度默认生成。
- 如果创建 `CLAUDE.md`，不要复制规则，只指向 `AGENT.md`，避免两份说明不一致。

## `README.md` 模板

```markdown
# <项目名>

## 项目概览
[用 2-3 句话说明这个项目要解决什么问题]

## 当前阶段
[术语统一 / 绘制路线图 / 明确问题 / 分解问题 / 优先排序 / 工作计划 / 分析论证 / 综合建议 / 方案呈现 / 已完成]

## 关键文件
- `brief.md`：项目总控
- `decision-map.md`：复杂项目路线图（如有）
- `0-inputs/`：原始资料（如有）
- `0-materials/`：处理后资料和证据（如有）
- `1-problem-definition/` ~ `7-communication/`：按需创建的七步法阶段产物

## 下一步
[一个最重要的下一步]
```

## `AGENT.md` 模板

```markdown
# <项目名> Agent Context

## 项目目标
[项目要达成的目标]

## 当前真实问题
[一句话定义真正要解决的问题]

## 工作规则
- 使用 structured-problem-solving 技能推进。
- 每轮最多问 1 个会改变分析路径的阻塞问题；不存在阻塞时声明假设并继续。
- 给确定性问题定义前盘点现有上下文、资料和缺口；没有外部资料时记录后继续。
- 如果使用资料目录，原始资料只放入 `0-inputs/`，处理后资料放入 `0-materials/`。
- 不要把资料缺口伪装成结论。
- 每次阶段推进后更新 `brief.md`。
- 如果存在 `decision-map.md`，它是路线状态的唯一来源。

## 下一步
先读取 `brief.md`。如果存在 `decision-map.md`，再读取它；否则按 `brief.md` 的当前阶段继续。
```

## `CLAUDE.md` 模板

```markdown
# <项目名>

本项目的 Agent 工作上下文统一维护在 [AGENT.md](AGENT.md)。

进入本项目工作时，先读取并遵循 `AGENT.md`；不要在 `CLAUDE.md` 中复制规则内容，避免两份说明不一致。
```

## `brief.md` 模板

```markdown
# <项目名>

## 当前阶段
[术语统一 / 绘制路线图 / 明确问题 / 分解问题 / 优先排序 / 工作计划 / 分析论证 / 综合建议 / 方案呈现 / 已完成]

## 真实问题
[一句话定义真正要解决的问题]

## 背景
[为什么现在要解决]

## 资料状态
- 原始资料：
- 已转换/已整理：
- 关键证据：
- 资料缺口：
- 术语表：`0-materials/domain-glossary.md`（如有）
- 路线图：`decision-map.md`（如有）

## 成功标准
- [可验证标准]

## 边界
- 范围内：
- 范围外：

## 当前假设
- [假设和置信度]

## 证据状态
- 已知道：
- 仍需验证：

## 决策日志
- YYYY-MM-DD：[关键决定；复杂项目只写摘要，细节指向 `decision-map.md` 或阶段文件]

## 下一步
[一个最重要的下一步]
```

创建项目时优先使用用户指定的名称。用户未指定时，基于已经确认或标记为临时的问题定义生成 1 个简短项目名，告知用户后继续；只有名称会影响目录归属或存在实质歧义时才询问。
