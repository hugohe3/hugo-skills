# Prompt And Agent Handoff

只有当最终执行者是 AI/agent 时，才把结构化问题规格转成 prompt、提示词或 agent 任务说明。不要为了“看起来完整”默认生成提示词。

## Principle

提示词可以复杂，但复杂性必须来自问题本身，而不是模板堆砌。复杂提示的目标是让 AI 稳定执行一个已经定义清楚的问题解决流程。

生成提示词前必须确认：

- 问题已经定义清楚。
- 已有资料、证据和资料缺口已盘点。
- 输入、输出、边界、成功标准明确。
- 需要 AI 执行的是一次性任务、长期 system prompt，还是多步骤 agent 工作流。

## Message Layer

消息层：

- **System prompt** - 长期身份、全局行为边界、工具规则和安全边界。
- **User prompt** - 当前任务、输入数据、即时问题和输出要求。
- **Assistant prefill** - 只在需要强制开头、XML/JSON 起始标记或角色话术时使用。
- **多轮 messages** - 用于提示链、二次检查、修订和工具结果回填。

选择规则：

- 长期复用、面向 agent 或团队工作流：优先拆成 `system` + `user`。
- 一次性任务：可以只给用户提示词，但仍要有清晰结构。
- 需要强格式输出：可用 assistant prefill 或明确输出起始标记。
- 需要复杂验证：用多轮 messages 或提示链，不要把所有动作塞进单轮。

## Anthropic Complex Prompt Components

复杂提示常见 10 个组件。不是每个提示都需要全部组件，但复杂任务通常应先用足组件跑通，再精简。

| 组件 | 作用 | 放置建议 |
|---|---|---|
| 角色 / 任务上下文 | 定义 AI 身份、目标、业务场景 | 靠前，常放 system |
| 语气上下文 | 控制沟通风格、读者关系、严肃度 | 靠前或并入角色 |
| 背景资料 | 放用户背景、业务背景、项目上下文 | 靠前或资料区 |
| 详细规则 | 定义必须做、禁止做、失败处理 | 靠前到中部 |
| 示例 | 展示理想输出、边界案例、正反例 | 规则后或输入前 |
| 输入数据 | 文档、代码、历史、检索结果、用户问题 | 用 XML 或标题清晰包裹 |
| 即时任务 | 重新说明当前要做什么 | 长提示靠后 |
| 思考 / 证据步骤 | 要求先分析、抽证据、检查条件 | 靠近最终任务 |
| 输出格式 | XML、JSON、Markdown、表格、代码块 | 靠后 |
| 预填充 | 强制输出开头或标签 | assistant 消息 |

默认正文顺序：

```text
任务/角色上下文
语气和行为规则
输入数据或资料边界
示例（如需要）
当前用户问题或即时任务
思考、证据或检查步骤（如需要）
输出格式
```

长文档和 RAG 场景可以把资料放前面，但最终任务、问题和输出格式应靠后。

## Structural Patterns

### XML Boundary Pattern

Claude 类模型尤其适合用 XML 标签区分指令、资料、示例和输出。

```text
<role>
你是{role}。
</role>

<context>
{background}
</context>

<source_materials>
{materials}
</source_materials>

<task>
{task}
</task>

<constraints>
- {constraint_1}
- {constraint_2}
</constraints>

<output_format>
{format}
</output_format>
```

### Evidence-First Pattern

适合事实、研究、战略判断、法律/医疗/金融等高风险分析。

```text
在回答前，先完成：
1. 从 <source_materials> 中提取与问题直接相关的证据。
2. 区分事实、假设和观点。
3. 标注资料缺口。
4. 只基于证据给出结论；无法确认时说明不确定。

最终输出只包含：
- 结论
- 关键证据
- 未验证假设
- 资料缺口
- 下一步验证建议
```

### Few-Shot Pattern

需要稳定格式、语气、分类边界或复杂判断时加入示例。示例质量优先于数量。

```text
<examples>
<example>
<input>...</input>
<ideal_output>...</ideal_output>
</example>
<example>
<input>边界或易失败案例</input>
<ideal_output>...</ideal_output>
</example>
</examples>
```

### Prompt Chain Pattern

当任务太复杂时拆成链路，不要塞进单一提示：

1. 定义问题。
2. 整理资料和证据。
3. 结构化拆解。
4. 提出假设。
5. 验证假设。
6. 综合建议。
7. 方案呈现。

每一步的输出作为下一步输入，并保留版本和证据来源。

## Agent Handoff Template

用于多步骤 agent 工作流，复杂一点是合理的。

```markdown
## System Prompt

你是{role}，负责帮助用户完成{problem_type}的结构化问题解决。

### Core Rules
- 先解决正确的问题，再设计答案。
- 每轮最多问 1 个会改变分析路径的阻塞问题；不存在阻塞时声明假设并继续，除非用户明确要求问题清单。
- 给确定性问题定义前盘点现有上下文、资料和缺口；没有外部资料时记录后继续。
- 区分事实、假设和观点。
- 不要把资料缺口伪装成结论。
- 只有用户要求持久化交付物或现有项目已落盘时，才使用 `projects/<YYYYMMDD>-<project-name>/` 项目结构。

### Mode Selection
- Quick：单次或低风险问题，留在对话中完成。
- Standard：需要拆解、证据判断或方案比较，但不创建项目文件。
- Project：明确需要持久化、可追溯交付物时，先创建最小 `brief.md`，再按需扩展。

### Workflow
按最轻可用模式选择必要步骤；合并或跳过已经满足的步骤，不要求用户逐阶段确认。

1. 复述真实问题。
2. 盘点资料：记录来源、证据和缺口；只有 Project 模式才按需归档、转换和建立索引。
3. 统一术语：澄清关键概念、命名和边界。
4. 明确问题：背景、成功标准、边界、限制、相关人、资源；资料不足时使用临时定义。
5. 绘制路线图：仅在存在多个未决分支时区分已决事项、开放问题和迷雾。
6. 分解问题：MECE 逻辑树、鱼骨图或多维图谱。
7. 优先排序：80/20、重要性-紧急性、影响-难度。
8. 按需制定工作计划：只有需要执行落地时才明确任务、时间、责任人/角色、资源和风险。
9. 分析论证：数据、根因、假设验证、反例检查。
10. 综合建议：方案比较、推荐方案、风险和取舍。
11. 方案呈现：结论先行、金字塔原则、故事线。

### Output Discipline
- 阶段中只输出当前阶段需要的内容。
- 每次推进后更新资料状态、证据状态和下一步。
- 用户确认前不要把草稿当最终结论。
```

## User Prompt Template

```markdown
<problem_context>
{用户的背景、目标、当前困惑}
</problem_context>

<available_materials>
{已获得资料清单、来源、可信度、摘要}
</available_materials>

<known_gaps>
{资料缺口、待验证问题}
</known_gaps>

<current_stage>
{术语统一 / 绘制路线图 / 明确问题 / 分解问题 / 优先排序 / 工作计划 / 分析论证 / 综合建议 / 方案呈现}
</current_stage>

<task>
请基于当前阶段完成{具体任务}。
</task>

<requirements>
- 每轮最多问 1 个会改变分析路径的阻塞问题；没有阻塞时声明假设并继续。
- 明确区分事实、假设和观点。
- 如资料不足，只输出临时结论并列出资料缺口。
- 输出必须能进入下一阶段。
</requirements>

<output_format>
{期望输出结构}
</output_format>
```

## Compact Handoff Template

适合一次性或轻量任务：

```markdown
## 适用问题
[这个提示词或 agent 任务解决的问题]

## 最终提示词 / 任务说明
[可直接使用的提示词或 agent 指令]

## 输入变量
| 变量 | 含义 | 示例 |
|---|---|---|

## 输出要求
[格式、结构、详细度、禁止项]

## 测试样例
1. [正常样例]
2. [边界样例]
3. [易失败样例]

## 评估标准
- [可检查的质量标准]
```

## Testing And Iteration

提示词交付必须附带测试样例。至少包含：

1. **正常样例** - 输入信息充分，预期能完整跑完流程。
2. **资料缺口样例** - 关键信息不足，预期应输出临时定义和缺口，而不是硬下结论。
3. **边界/干扰样例** - 用户给出表面任务或带偏见资料，预期应识别真实问题和证据限制。

评估标准：

- 是否只提出会改变分析路径的阻塞问题，且每轮不超过一个。
- 是否先盘点现有上下文、资料和缺口再给确定性问题定义。
- 是否区分事实、假设和观点。
- 是否没有把资料缺口伪装成结论。
- 是否按麦肯锡七步法推进到正确阶段。
- 输出是否能支持下一步行动。

如果测试失败，只改最小必要部分，并标记为 v2、v3。不要每次失败都重写整套提示词。

## Checks

- 提示词是否服务于已经定义清楚的问题？
- 输入、输出、边界、成功标准是否完整？
- 是否保留资料来源和证据要求？
- 是否明确失败处理：提问、说明不确定、降级输出、拒绝或请求更多资料？
- 是否有测试样例覆盖正常、边界和易失败场景？
- 是否保留了复杂任务需要的消息层、组件层和验证链路？
