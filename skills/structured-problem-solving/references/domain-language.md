# Domain Language

结构化问题解决必须先有稳定语言。用户、材料或团队对同一个词的理解不一致时，后续拆解、证据和建议都会漂移。

## 触发条件

出现以下情况时先处理术语：

- 用户使用“平台、系统、客户、用户、项目、流程、方案”等宽泛词，但含义不清。
- 不同材料对同一概念使用不同词，或同一词指向不同对象。
- 方案讨论依赖概念边界，例如谁是决策者、谁是使用者、什么算完成、什么算失败。
- 需要把分析结果交给团队、AI agent 或后续会话继续执行。

## 工作规则

1. **挑战冲突**
   - 如果新说法和已定义术语冲突，立即指出冲突。
   - 用“你这里的 X 是指 A 还是 B？”来迫使边界清晰。

2. **收敛命名**
   - 为关键概念提出一个规范名称。
   - 记录应避免的同义词或歧义词。
   - 不为了好看而重命名已有稳定术语。

3. **用场景压测**
   - 构造 1-2 个具体边界场景，检查术语是否仍然成立。
   - 如果场景推翻当前定义，先修正定义，再继续分析。

4. **交叉核对**
   - 用户说法、原始材料、已有文档、代码或数据不一致时，标记为待验证。
   - 不把术语分歧伪装成事实结论。

5. **即时记录**
   - 一旦术语被确认，立即写入项目材料。
   - 不等到最后统一整理，因为术语会影响后续每一步。

## 术语表位置

需要落盘时，在 `0-materials/domain-glossary.md` 维护术语表：

```markdown
# Domain Glossary

| 术语 | 定义 | 避免混用 | 边界 / 例子 | 来源 |
|---|---|---|---|---|
| [术语] | [一句话定义] | [不要叫成什么] | [包含什么 / 不包含什么] | [材料或用户确认] |
```

只记录会影响问题定义、拆解、决策或交付的术语。不要把普通词典定义、实现细节或临时想法塞进术语表。

## 与决策记录的边界

`domain-glossary.md` 只记录术语和概念边界，不记录方案取舍。

如果术语澄清导致了真实决策：

- 轻量项目把决策摘要写入 `brief.md` 的“决策日志”。
- 复杂项目把决策写入 `decision-map.md`，并指向对应阶段文件。

不要在术语表里维护第二份决策记录。
