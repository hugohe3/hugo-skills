# Decision Map

当问题太大、路径不清或需要跨多轮推进时，先建立决策地图。地图是索引，不是资料仓库；它只记录当前路线、已决事项、开放问题和迷雾，并指向真正承载细节的文件。

`decision-map.md` 是复杂项目的路线状态唯一来源。不要在 `brief.md`、`AGENT.md` 或阶段文件中重复维护完整路线。

## 使用原则

- **一次只解决一个问题** - 每轮选择一个最重要的开放问题推进，不要并行展开多个分支。
- **名称优先** - 开放问题和决策用可读标题命名，不用裸编号代替。
- **地图只做索引** - 详细论证放在阶段文件、研究笔记、原型或资料文件中。
- **先声明迷雾** - 看得见但说不清的问题留在迷雾区，不强行拆成任务。
- **随着推进更新地图** - 每解决一个问题，都更新已决事项、开放问题和迷雾。
- **不替代工作计划** - 地图管理未决问题；`4-work-planning/` 管已经决定要执行的任务、时间、责任和风险。

## 地图位置

需要落盘时，在项目根目录创建 `decision-map.md`：

```markdown
# Decision Map

## Notes
[项目背景、必须遵守的技能、偏好和约束]

## Decisions So Far
- [决策标题](相对路径) — [一句话说明结论]

## Open Questions
- [问题标题](相对路径或待建文件) — 类型：[research / prototype / grilling / task]；状态：[open / blocked / resolved]

## Fog
- [尚不能精确表述的风险、待观察区域或后续可能问题]
```

`brief.md` 继续作为总控文件；`decision-map.md` 只管理路线和问题队列。

## 问题类型

| 类型 | 适用场景 | 完成标准 |
|---|---|---|
| research | 需要外部资料、官方文档、数据或已有材料才能回答 | 有带来源的研究摘要和证据缺口 |
| prototype | 需要一个低成本草图、样例、流程、界面或代码片段帮助判断 | 有可被评价的粗原型和明确反馈点 |
| grilling | 需要用户决策、偏好或业务判断 | 通过一问一答形成可记录的结论 |
| task | 必须完成的前置动作，无需继续讨论，但会阻塞后续判断 | 动作完成，并记录由此产生的新事实 |

默认类型是 `grilling`，但如果答案能通过资料或仓库探索获得，改为 `research` 或直接自行探索。不要把已确定的执行任务放进 `Open Questions`；它们属于 `4-work-planning/`。

## 迷雾与开放问题

判断标准不是“能否回答”，而是“能否精确提问”：

- 能用一句话说清要解决什么，进入 `Open Questions`。
- 只能模糊描述风险、方向或待观察区域，放入 `Fog`。
- 当一个问题被解决后，检查是否有迷雾可以升级为新的开放问题。

不要预先把迷雾切成一堆看似完整的任务；这会制造伪确定性。

## 解决流程

1. 读取 `brief.md` 和 `decision-map.md`。
2. 选择一个最高优先级、未阻塞的开放问题。
3. 根据类型读取材料、做原型、追问用户或完成前置任务。
4. 把结论写入对应阶段文件。
5. 在 `decision-map.md` 中把该问题标为 `resolved`，并在 `Decisions So Far` 增加一行指针。
6. 新出现的问题进入 `Open Questions`，仍模糊的内容留在 `Fog`。
7. 如果结论已经转化为执行安排，把任务写入 `4-work-planning/`，地图只保留指向该计划的决策摘要。
