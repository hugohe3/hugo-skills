# Structured Problem Solving

## Overview

用麦肯锡七步问题解决法作为主流程，把模糊想法、复杂困惑或战略问题推进为清晰的问题定义、MECE 分析框架、可验证分析、综合建议和可沟通方案。提示词只是最终交付形式之一；核心目标是帮助用户解决正确的问题。

## Hard Gates

```text
NO FINAL ANSWER BEFORE THE PROBLEM IS FRAMED
NO PROBLEM DEFINITION WITHOUT MATERIAL INVENTORY
ONE CLARIFYING QUESTION PER TURN
DISCOVER BEFORE ASKING
FOG IS NOT A QUESTION UNTIL IT CAN BE STATED
```

- 在问题尚未定义清楚前，不要直接给最终建议、完整方案、提示词或执行计划。
- 进入“明确问题”前，必须先知道已获取哪些资料、资料可信度如何、还缺哪些资料。
- 每轮只问 1 个最关键问题，禁止一次性问完所有问题，除非用户明确要求问题清单。
- 如果问题可通过读取材料、检索仓库或检查已有上下文回答，先自行探索，不要把可查事实抛给用户。
- 模糊风险、待观察区域和未成形判断先放入“迷雾”，只有能被精确表述为一个待解决问题时，才进入当前问题队列。

## When to Use

使用本技能：

- 用户只描述了焦虑、困惑、想法或表面任务，真实问题尚不清楚。
- 用户需要做战略分析、业务拆解、研究判断、职业规划、复杂任务规划或决策建议。
- 用户要求“帮我分析问题”“定义问题”“拆解问题”“提炼假设”“形成结论”“写给 AI 的任务说明”。
- 用户想写提示词，但真实目标是让 AI 稳定解决某类问题或执行某个工作流。
- 用户提供长材料，希望吸收其方法论并形成可复用技能、流程或 agent 行为。
- 用户的问题很大、路径不清，可能需要跨多轮拆成研究、原型、追问或执行任务。

不要用于：用户只要求执行明确、机械、低歧义任务；或问题已经有完整规格，当前只需要代码实现、文件转换或简单编辑。

## Core Checklist

必须按顺序推进，除非用户明确要求跳到某一步：

1. **复述真实问题** - 区分表面请求和背后的决策/行动问题。
2. **逐轮澄清** - 每轮只问 1 个最关键问题；每个问题都给出推荐答案或候选答案。
3. **判断落盘模式** - 简单问题留在对话；跨多轮问题创建最小项目框架；复杂问题再扩展阶段目录。
4. **整理资料** - 归档原始材料，必要时转换为 Markdown；默认轻量证据文件，复杂项目再拆分来源索引、资料盘点和缺口。
5. **统一术语** - 发现模糊、冲突或重名概念时，先建立共享语言，再继续分析。
6. **绘制路线图** - 只有问题大到不能一次完成时，建立决策地图，区分已决事项、当前问题和迷雾。
7. **明确问题** - 先盘点已获取资料和资料缺口，再定义问题、范围、利益相关者和紧急程度。
8. **分解问题** - 用 MECE 逻辑树、鱼骨图或其他框架拆解。
9. **优先排序** - 用 80/20、重要性-紧急性或影响-难度矩阵聚焦关键问题。
10. **工作计划** - 明确分析任务、时间表、责任人、资源和风险。
11. **分析论证** - 收集数据、分析原因、验证假设。
12. **综合建议** - 形成、评估并选择解决方案。
13. **方案呈现** - 用金字塔原则、故事线和可视化进行沟通。
14. **按需转化** - 如果最终执行者是 AI/agent，再生成提示词或任务说明。

## Interaction Pattern

协作阶段默认使用短格式：

```markdown
我当前理解：
[用 1-3 句话复述真实问题]

关键缺口：
[当前最影响分析质量的缺口]

我的问题：
[只问 1 个问题；优先给 2-4 个选项]

我的推荐：
[给出推荐答案和理由；如果答案应由用户决定，说明推荐默认值]
```

信息足够后，先给 2-3 种分析路径，而不是直接给唯一答案。

```markdown
## 当前阶段
[术语统一 / 绘制路线图 / 明确问题 / 分解问题 / 优先排序 / 工作计划 / 分析论证 / 综合建议 / 方案呈现]

## 推荐分析路径
[推荐路径和理由]

## 可选路径
1. [路径 A：适用场景和取舍]
2. [路径 B：适用场景和取舍]
3. [路径 C：如需要]

## 需要你确认
[只问 1 个最关键问题，并给出推荐答案]
```

## Quality Gate

交付前自检：

- 真实问题是否不同于表面请求？
- 明确问题前是否完成资料盘点？
- 是否先探索了可自行查证的信息，而不是把可查事实问给用户？
- 是否列出资料缺口和后续补充路径？
- 关键术语是否定义一致，是否避免同义词漂移？
- 大问题是否区分了已决事项、当前问题和迷雾？
- 问题是否一句话能说清？
- 拆解是否符合 MECE？
- 是否完成优先排序并说明取舍？
- 是否有工作计划和风险应对？
- 是否有明确假设、证据缺口和验证方法？
- 是否区分了事实、假设和观点？
- 最终交付是否能支持决策、行动或稳定执行？
